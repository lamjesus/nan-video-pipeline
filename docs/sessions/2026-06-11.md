# Sesión 2026-06-11 — Rescate del template de render + guion validado (Tarea D)

Bitácora de lo cambiado en esta PR y por qué. Memoria para el equipo.

## Qué se hizo

- **Rescate de la PR #4.** La PR #4 (template HTML + GSAP + captions) se mergeó
  contra `feat/compose-manifest-and-workspace` 21 segundos **después** de que esa
  rama entrara en `main` vía PR #3 → `src/render/` nunca llegó a `main`. Esta
  rama parte de la rama original de adri, le mergea `main` (el merge trajo la
  etapa de subtítulos, que faltaba allí) y reconcilia el `yarn.lock`. Moraleja
  para PRs apiladas: re-apuntar la base a `main` antes de mergear.
- **Tarea D — Guion con qwen3.6.** `src/pipeline/script-util.ts` (lógica pura):
  `extractJson` tolera bloques `<think>`, vallas markdown y prosa alrededor;
  `validateStoryboard` exige 10 escenas, campos requeridos y tiempos coherentes,
  con errores que nombran el campo (`scenes[3].voiceover`) para usarlos como
  feedback. `01-script.ts`: retry con ese feedback (3 intentos), `reasoning_config`
  desactivado con degradación si el cluster lo rechazara, slug opcional
  (`yarn script "<tema>" [slug]`), `createNanCall` y errores ERROR/WHY/FIX.
- **Casos generados sin registro manual.** `load.ts` hace fallback a un import
  dinámico (`src/content/<slug>.ts`) con el formato del slug validado; `CASES`
  queda para los casos curados. Se commitea `caso-alejandria` como segundo caso
  de ejemplo cargable.
- **Fix Windows en `load.ts`.** El guard de auto-ejecución comparaba
  `import.meta.url` con `file://${argv[1]}`, que en Windows nunca coincide
  (backslashes) → `yarn load` salía en silencio sin probar nada. Ahora usa
  `pathToFileURL`.

## Verificación

- `yarn typecheck` ✅ · `yarn test` ✅ (14 tests nuevos de `script-util`).
- **E2E real contra el cluster:** `yarn script "La biblioteca de Alejandría"
  caso-alejandria` → Storyboard válido al primer intento (10 escenas, 45s,
  ~26s de generación). El cluster **acepta `reasoning_config`** (sin warning).
- `yarn load caso-alejandria` carga por fallback; `yarn load caso-inexistente`
  falla con mensaje claro y pista de generación.

## Hallazgos

- PRs apiladas: si la base se mergea primero, GitHub no re-apunta la PR hija a
  `main` automáticamente → revisar `baseRefName` antes de mergear.
- `yarn load` era un no-op en Windows (guard de `import.meta.url`).
- `caso-ejemplo` tiene 9 escenas; la validación de 10 aplica solo a casos
  generados nuevos, no al cargador.
